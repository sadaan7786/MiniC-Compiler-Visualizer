Quick Samples:

Phase 1 — Lexical Analysis

Token stream generated by Flex
How it works: The lexer (Flex) scans source code character by character, groups characters into lexemes, and classifies each as a token type (KEYWORD, IDENTIFIER, INT_CONST, OPERATOR, etc.).
# Token Type Lexeme Line

Phase 2 — Syntax Analysis & AST

Parse tree built by Bison (LALR(1) grammar)
How it works: Bison uses the grammar rules (parser.y) to verify the token stream forms a valid program. As it recognises rules, it builds an Abstract Syntax Tree — a tree where each node is an operation and leaves are operands.
Drag to pan · Scroll to zoom

Phase 3 — Semantic Analysis & Symbol Table

Variable declarations, types, and scopes
How it works: The semantic analyser checks for undeclared variables, duplicate declarations, and type mismatches. All identifiers are stored in the Symbol Table — a dictionary keyed on name.
Name Type Scope Declared Line Initialized

Phase 4 — Intermediate Code (Three Address Code)

Machine-independent intermediate representation
How it works: Complex expressions are broken into simple 3-operand instructions: result = op1 OP op2. Temporaries (t1, t2…) store intermediate values. Control flow uses labels and conditional jumps.

Phase 5 — Code Optimization

Constant folding + dead code elimination
Optimizations applied:
  • Constant Folding: Evaluate constant expressions at compile time (e.g. 3 * 2 → 6)
  • Dead Code Elimination: Remove temporaries whose values are never used after assignment (marked with [DEAD])

Phase 6 — Target Code Generation

Pseudo x86-style assembly output
How it works: Each TAC instruction maps to one or more assembly instructions. Registers R0 (accumulator), R1, R2 hold temporary values. MOV, ADD, SUB, MUL, CMP, JE, JMP and CALL mirror real x86 semantics.
